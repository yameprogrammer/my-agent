# 프론트엔드 재구현 작업 계획서 (Frontend Rebuild Plan)

> **작성일**: 2026-07-13  
> **근거 문서**: [frontend_rebuild_review.md](./frontend_rebuild_review.md)  
> **목표**: Streamlit 프론트엔드를 Vite + Vanilla JS SPA로 완전 교체하여 반응성, 퍼포먼스, UI/UX를 프로덕션 수준으로 향상시킨다.

## 1. 작업 개요

### 1.1 스프린트 편성

본 작업은 기존 스프린트 보드의 **Sprint 6 (프론트엔드 재구현)**으로 편성한다.

| 단계 | 코드 | 목표 | 예상 난이도 |
| :--- | :--- | :--- | :---: |
| 프로젝트 초기화 & 디자인 시스템 | **6-A** | Vite 프로젝트 셋업 + CSS 디자인 토큰 + 공통 컴포넌트 | ⭐⭐ |
| 인프라 레이어 | **6-B** | API 클라이언트 + JWT 인증 + WebSocket 매니저 + SPA 라우터 | ⭐⭐ |
| 인증 & 대시보드 | **6-C** | 로그인/회원가입 + 프로젝트 대시보드 | ⭐⭐ |
| 프로젝트 상세 (CRUD 탭) | **6-D** | AI 기획 파트너 + 세계관 + 캐릭터 + 회차 관리 + 설정 | ⭐⭐⭐ |
| 실시간 집필 모니터 | **6-E** | WebSocket 스트리밍 + HITL 피드백 + AI 평가 리포트 | ⭐⭐⭐⭐ |
| FastAPI 정적 서빙 통합 | **6-F** | 빌드 산출물 FastAPI 마운트 + 프로덕션 설정 | ⭐⭐ |
| 통합 테스트 & 마무리 | **6-G** | E2E 시나리오 테스트 + 반응형 검증 + 성능 프로파일링 | ⭐⭐⭐ |

### 1.2 전제 조건

- 백엔드 API 코드(`app/`)는 **변경하지 않는다** (CORS `["*"]` 이미 설정됨)
- 기존 `ui/` 디렉토리는 **보존**하며, 새로운 `frontend/` 디렉토리에 구현한다
- 개발 중에는 Vite dev server의 프록시를 통해 백엔드(`:8080`)와 통신한다
- 빌드 후에는 `frontend/dist/`를 FastAPI `StaticFiles`로 서빙한다

### 1.3 에이전트 수행 가이드라인

1. 각 하위 단계(6-A ~ 6-G) 완료 시 **반드시 동작 검증**을 수행한 후 다음 단계로 넘어간다.
2. API 연동 시 **백엔드가 실행 중이지 않아도** 프론트엔드 단독으로 빌드·번들링이 정상 완료되는지 확인한다.
3. CSS 작업 시 **다크모드 + 라이트모드** 양쪽 모두 확인한다.
4. 반응형 디자인은 **Galaxy Z Fold 4 해상도(2176×1812 언폴드 / 904×2316 폴드)**를 기준으로 검증한다.

## 3. 작업 순서 및 의존관계

```

                                                                                    │
                                                                                    ↓
                                                                          6-E (집필 모니터)
                                                                                    │

                                                                          6-F (FastAPI 통합)
                                                                                    │
                                                                                    ↓
                                                                          6-G (통합 테스트)
```

- **6-E → 6-F**: 전체 기능 구현 완료 후 빌드 통합
- **6-F → 6-G**: 통합 빌드 후 최종 검증

## 5. 리스크 및 완화 방안

| 리스크 | 영향 | 완화 방안 |
| :--- | :--- | :--- |
| WebSocket 재연결 시 상태 동기화 실패 | 집필 모니터 데이터 유실 | `current_state` 이벤트로 전체 상태 복구 + 로컬 드래프트 캐싱 |
| Vanilla JS로 복잡한 UI 상태 관리 한계 | 코드 복잡도 증가 | 경량 옵저버 패턴 + 컴포넌트 단위 상태 캡슐화 |
| Vite 빌드와 FastAPI 라우팅 충돌 | SPA 라우팅 실패 | `index.html` fallback 설정으로 모든 비-API 경로를 SPA로 포워딩 |
| Galaxy Z Fold 4 특수 해상도 대응 | 레이아웃 깨짐 | 폴드/언폴드 미디어 쿼리 + CSS Grid 기반 유동 레이아웃 |

## 6. 성공 기준

1. **반응 속도**: 모든 페이지 전환이 100ms 이내
2. **스트리밍 지연**: WebSocket 텍스트 스트리밍이 수신 즉시 렌더링 (체감 지연 없음)
3. **번들 크기**: 프로덕션 빌드 100KB 이하 (gzip 기준)
4. **반응형**: Galaxy Z Fold 4 폴드/언폴드 + 일반 모바일/태블릿/데스크톱 모두 정상
5. **기능 완전성**: 현재 Streamlit UI의 모든 기능이 동등 이상으로 구현됨
6. **단일 서버**: FastAPI 하나로 API + 프론트엔드 모두 서빙 (Streamlit 서버 제거)
