# 외부 메모리 및 RAG 설계

## 목적

장편 웹소설의 가장 큰 문제는 누적 맥락을 잃는 것이다. 따라서 시스템은 모든 주요 서사 정보를 외부 메모리에 저장하고, 단계별 작업 시 필요한 맥락만 선택적으로 검색해 사용해야 한다.[cite:11][cite:20][cite:22]

## 메모리 계층

메모리는 세 가지 계층으로 나눈다.

### 1. Canonical Memory

작품의 공식 설정이다. 승인된 정보만 저장한다.

- 작품 주제
- 장르/톤
- 세계관 규칙
- 인물 프로필
- 세력 정보
- 능력 체계
- 고정 사건
- 결말 방향

### 2. Operational Memory

진행 중인 작업 상태다.

- 현재 아크 목표
- 최근 5화 요약
- 최근 20화 누적 변화
- 열려 있는 떡밥 목록
- 예정된 회수 포인트
- 캐릭터 감정 상태
- 독자 반응 시뮬레이션 결과

### 3. Retrieval Memory

RAG 검색용 문서 묶음이다.

- 화별 요약
- 장면별 요약
- 대사 패턴 예시
- 장르 레퍼런스 규칙
- 금지 표현/문체 규칙

## 저장 방식

구조화 데이터는 PostgreSQL 테이블에 저장한다. 서술형 요약과 장면 문서 등은 문서 테이블 + 벡터 인덱스로 저장한다. 메타데이터 필터로 `novel_id`, `arc_id`, `episode_range`, `character_id`, `memory_type`, `status`를 둔다.

## 검색 전략

각 에이전트는 동일한 메모리를 모두 받지 않는다. 역할별 검색 계약(contract)을 둔다.

예시:

- 전체 설계 AI: 주제 후보, 타깃 장르 규칙, 독자 기대치만 검색
- 아크 설계 AI: 작품 목표, 결말 방향, 인물 핵심 욕망, 세계관 제약 검색
- 화 세부 설계 AI: 해당 아크 요약, 최근 3화 요약, 관련 복선, 참여 캐릭터 상태 검색
- 집필 AI: 이번 화 비트시트, 직전 1~2화 상세 요약, 말투 규칙, POV 규칙 검색

## 문서 청킹 전략

청킹은 서사 단위로 해야 한다. 단순 토큰 길이 기준 청킹은 피한다.

- 작품 개요: 1 문서
- 아크 개요: 1 아크 = 1~3 문서
- 화 요약: 1 화 = 1 문서
- 장면 요약: 1 장면 = 1 문서
- 캐릭터 카드: 1 인물 = 1 문서
- 복선 스레드: 1 스레드 = 1 문서

## 요약 계층

긴 소설 운영을 위해 요약은 계층형으로 유지한다.

- short summary: 최근 1~3화 상세
- medium summary: 최근 아크 요약
- long summary: 작품 전체 진행 요약

장편 생성 연구에서 스토리 히스토리를 현재 이벤트 기준으로 동적으로 압축해 사용하는 방식이 일관성 유지에 유효한 것으로 제시되었다.[cite:22]

## 메모리 갱신 규칙

모든 승인된 산출물 후에 메모리를 갱신한다.

- 새로운 인물 등장 -> character state 생성
- 복선 심기 -> thread 상태 `planted`
- 복선 강화 -> thread 상태 `reinforced`
- 복선 회수 -> thread 상태 `resolved`
- 능력/관계 변화 -> timeline event 기록

메모리 갱신은 자동 제안 후 인간 승인 모드로 시작하는 것이 안전하다.
